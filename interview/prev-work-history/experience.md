# 1. Kong плагін для soft‑authentication та rate limiting

Однією з задач було оптимізувати обробку трафіку в системі маркетплейсу.

## Контекст

Уся зовнішня взаємодія з бекендом проходила через **Kong API Gateway**, який виступав єдиною точкою входу в систему. Через нього маршрутизувався трафік до великої кількості сервісів.

При цьому різні типи користувачів повинні були мати **різні обмеження по rate limiting**.

## Проблема

Rate limiting залежав від того, **авторизований користувач чи ні**.

Якщо реалізовувати цю логіку безпосередньо в сервісах, виникали кілька проблем:

* кожен сервіс мав би реалізовувати власну логіку авторизації
* збільшувалося навантаження на backend
* ускладнювалась архітектура

Тому було вирішено перенести визначення типу користувача **якнайраніше в потоці запиту — на рівень gateway**.

## Рішення

Я спроєктував і реалізував **кастомний Kong plugin на Lua**, який виконує soft‑authentication.

Його основна ідея — спробувати ідентифікувати користувача, але не блокувати запит, якщо авторизація відсутня.

Плагін працює так:

1. читає JWT із cookie
2. верифікує токен за допомогою public key
3. витягує `account_id`
4. якщо токен валідний — призначає authenticated consumer
5. якщо токен відсутній або невалідний — призначає anonymous consumer

Для анонімних користувачів додатково виконується **fingerprinting запиту на основі HTTP‑заголовків**, що дозволяє стабільніше застосовувати rate limiting.

## Результат

У результаті ми змогли:

* визначати тип користувача **ще на рівні gateway**
* застосовувати різні політики **rate limiting**
* зменшити навантаження на backend‑сервіси
* спростити логіку самих сервісів

Цей проєкт був більше інфраструктурним, а далі я хочу розповісти про приклад системи, яка безпосередньо обробляла бізнес‑транзакції.

# 2. Система продажу первинних активів через веб

Одним з найбільш цікавих проєктів для мене була розробка системи продажу **первинних ігрових активів через веб**.

## Контекст

До цього значна частина продажів відбувалась через мобільні платформи, такі як **Apple App Store** і **Google Play**.

Ці платформи беруть суттєву комісію з кожної транзакції, тому компанія хотіла створити можливість продавати активи **напряму через веб**.

## Проблема

Потрібно було спроєктувати систему покупки, яка:

* контролює **обмеження на покупку для користувача**
* інтегрується з платіжною системою
* гарантує **консистентність між сервісами і грою**
* коректно обробляє асинхронні відповіді від платіжних систем

## Архітектура рішення

Система складалась з двох основних сервісів.

### PS‑Store

Цей сервіс відповідав за зберігання транзакцій.

Використовувався **MongoDB**, де створювався запис покупки і зберігався її стан. Для критичних операцій застосовувались **транзакції MongoDB**, щоб гарантувати консистентність даних.

### PS‑Processor

Другий сервіс відповідав за **оркестрацію процесу покупки**.

Для цього використовувався **Temporal workflow**, який керував життєвим циклом транзакції.

## Потік покупки

Процес виглядав так:

1. користувач ініціює покупку
2. створюється запис транзакції в PS‑Store
3. запускається Temporal workflow
4. workflow перевіряє обмеження на покупку
5. відправляється запит до платіжної системи
6. система очікує результат транзакції
7. після успішної оплати відправляється повідомлення в гру
8. гра нараховує актив користувачу

Temporal дозволив зручно обробляти:

* retry
* асинхронні відповіді
* консистентність станів транзакції

## Результат

Ця система дозволила перенести продаж **з мобільних платформ на веб**, що значно зменшило комісію платформ і створило новий канал доходу для компанії.

Паралельно з цим я працював і над внутрішніми платформами, які використовувались гейм‑студіями для керування своїми проєктами.

# 3. Studio Portal та система авторизації

Ще одним напрямком роботи був **Studio Portal** — веб‑застосунок для гейм‑студій.

## Контекст

Через нього студії могли:

* керувати своїми організаціями
* налаштовувати ігрові тайтли
* працювати з блокчейн‑контрактами

Backend складався з великого CRUD‑сервісу з багатьма пов’язаними сутностями.

## Проблема

Основним викликом була побудова **гнучкої системи авторизації**, тому що:

* різні студії мають різні ролі
* різні ролі мають різні права доступу
* система повинна підтримувати кілька типів авторизації

## Рішення

Я використав **декларативний підхід до опису авторизації**.

Ідея полягала в тому, що:

* scopes і типи авторизації описуються **безпосередньо в protobuf сервісах**
* ці описи автоматично потрапляють у **OpenAPI**
* OpenAPI використовується для генерації клієнта для React застосунку

Для перевірки доступів використовувався **Casbin**, який виконував:

* матчинґ ролей
* перевірку дозволів

Самі ролі і доступи могли налаштовуватись через Studio Portal.

## Результат

Такий підхід дозволив:

* тримати правила авторизації **поруч з API контрактами**
* синхронізувати backend і frontend через OpenAPI
* зробити систему гнучкою для різних студій
