# Go Concurrency — Complete Guide

Пов'язані нотатки: [[interview/golang/0. Go Runtime Internals]]

---

## Part 1: Goroutines & Scheduler

### Goroutine

**Goroutine** — це легка одиниця виконання, якою керує **Go runtime**, а не операційна система.

Запуск:

```go
go processEvent(e)
```

#### Чому goroutines дешеві

| Thread (OS)  | Goroutine            |
| ------------ | -------------------- |
| ~1 MB stack  | ~2 KB stack          |
| керується OS | керується Go runtime |

Stack goroutine **динамічно росте і зменшується**.

Тому можна запускати **десятки тисяч goroutines**.

---

### Go Scheduler

**Scheduler** — це механізм runtime, який вирішує:

- яка goroutine виконується
- яка чекає
- коли переключити виконання

```
багато goroutines
↓
менше потоків
↓
CPU
```

---

### GMP Model

```
G — goroutine
M — machine (OS thread)
P — processor
```

Інтуїтивна модель:

```
goroutine = задача
thread = робітник
P = дозвіл виконувати Go код
scheduler = диспетчер
```

Схема виконання:

```
goroutines
   ↓
run queue
   ↓
scheduler
   ↓
threads
   ↓
CPU
```

---

### GOMAXPROCS

`GOMAXPROCS` — максимальна кількість goroutines, які можуть реально виконуватись **одночасно**.

```
GOMAXPROCS = 8 → максимум 8 goroutines паралельно
```

---

### Concurrency vs Parallelism

**Concurrency** — система може працювати з багатьма задачами, перемикаючись між ними.

```
CPU → taskA → taskB → taskC → taskA → ...
```

**Parallelism** — задачі реально виконуються одночасно на різних ядрах.

```
CPU core 1 → taskA
CPU core 2 → taskB
CPU core 3 → taskC
```

Go забезпечує concurrency; parallelism — автоматично, якщо є кілька ядер.

Mental model:

```
Concurrency:  один кухар, багато замовлень, перемикається між ними
Parallelism:  багато кухарів, кожен готує свою страву одночасно
```

---

### Що відбувається коли goroutine блокується

```go
resp, _ := http.Get(url)
```

```
goroutine блокується
↓
thread звільняється
↓
scheduler запускає іншу goroutine
```

Це робить Go ефективним для I/O workloads.

---

### Runtime Preemption

**Preemption** — runtime може **перервати goroutine**, навіть якщо вона не блокується.

Це запобігає монополізації CPU однією goroutine.

```
runtime ставить preemption flag
↓
goroutine призупиняється (стан зберігається у G struct)
↓
scheduler запускає іншу
↓
пізніше G продовжується з того ж місця
```

Стан goroutine (G struct):

```
G struct
 ├─ stack
 ├─ registers
 ├─ program counter
 └─ scheduler metadata
```

**До Go 1.14** preemption відбувався лише при блокуванні або syscall.
**Після Go 1.14** — async preemption, scheduler справедливіший.

---

### Небезпека unbounded goroutines

```go
// ❌ Погано
for i := 0; i < 1_000_000; i++ {
    go work()
}
```

```
1M goroutines → memory росте → scheduler overhead → latency деградує
```

Причинно-наслідковий ланцюжок (slow external API):

```
slow API
↓
goroutines живуть довше
↓
goroutines накопичуються
↓
росте пам'ять і latency
```

Production правило: завжди **bounded concurrency**.

## Part 2: Channels

### Що таке Channel

**Channel** — механізм комунікації між goroutines без спільного доступу до пам'яті.

Головна ідея Go:

```
Do not communicate by sharing memory;
share memory by communicating.
```

```go
ch := make(chan int)

ch <- 42      // send
v := <-ch     // receive
```

---

### Unbuffered vs Buffered

**Unbuffered:**

```go
ch := make(chan int)
// send блокується поки хтось не зробить receive
```

**Buffered:**

```go
ch := make(chan int, 5)
// send не блокується поки buffer не заповнений
```

Buffered channel = черга. Producer і consumer можуть працювати з різною швидкістю.

---

### Channel Direction

```go
func producer(ch chan<- int)   // send only
func worker(jobs <-chan Job)   // receive only
```

Робить API зрозумілішим і запобігає помилкам.

---

### Closing Channels

```go
close(ch)

// перевірка закриття
value, ok := <-ch  // ok = false → канал закритий

// ітерація до закриття
for job := range jobs {
    process(job)
}
```

**Правило:** канал закриває той, хто **пише** (producer).

#### Single Producer

```go
func producer(out chan<- int) {
    for i := 0; i < 10; i++ {
        out <- i
    }
    close(out)
}
```

#### Multiple Producers — WaitGroup

```go
var wg sync.WaitGroup

for i := 0; i < 3; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        produce(ch)
    }()
}

go func() {
    wg.Wait()
    close(ch)
}()
```

#### Common Closing Bugs

```
double close      → panic: close of closed channel
send after close  → panic: send on closed channel
close by receiver → ніколи не робити
```

---

### Select

```go
select {
case msg := <-ch1:
    handle(msg)
case msg := <-ch2:
    handle(msg)
}
```

Select = switch для каналів. Runtime вибирає готовий case.

Non-blocking select:

```go
select {
case msg := <-ch:
    handle(msg)
default:
    // нічого не готово — не блокуємось
}
```

---

### Channel Patterns

**Fan-out** — одна черга, багато workers:

```go
jobs := make(chan Job)
for i := 0; i < 10; i++ {
    go worker(jobs)
}
```

**Fan-in** — багато goroutines, один канал результатів:

```go
results := make(chan Result)
for _, job := range jobs {
    go func(job Job) {
        results <- process(job)
    }(job)
}
```

**Pipeline** — обробка розбита на етапи:

```go
func stage1(out chan<- Data)
func stage2(in <-chan Data, out chan<- Result)
```

**Channel як Semaphore:**

```go
sem := make(chan struct{}, 10)
sem <- struct{}{}    // acquire
<-sem                // release
```

---

### Common Channel Bugs

```
Deadlock:        send без receive — runtime виявить
Double close:    panic
Goroutine leak:  goroutine чекає канал, який ніколи не закриється
```

## Part 3: Controlling Concurrency

### Worker Pool

Worker pool — кількість goroutines **фіксована**, задачі стоять у черзі.

```
tasks
 ↓
queue (channel)
 ↓
workers (fixed goroutines)
 ↓
execution
```

```go
jobs := make(chan Job)

for i := 0; i < 10; i++ {
    go worker(jobs)
}

for _, job := range jobList {
    jobs <- job
}

func worker(jobs <-chan Job) {
    for job := range jobs {
        process(job)
    }
}
```

Коли використовувати:

```
Kafka consumers
job queues
background workers
batch pipelines
```

---

### Semaphore Pattern

Semaphore обмежує **одночасний доступ до ресурсу**.

```
goroutines
 ↓
semaphore (limit)
 ↓
execution
```

```go
sem := make(chan struct{}, 10)

for _, job := range jobs {
    sem <- struct{}{}

    go func(job Job) {
        defer func() { <-sem }()
        process(job)
    }(job)
}
```

```
1000 goroutines створюються
10 виконуються
990 чекають semaphore
```

Коли використовувати:

```
HTTP fan-out
DB connection limiting
external API calls
file operations
```

---

### Worker Pool vs Semaphore

| | Worker Pool | Semaphore |
|---|---|---|
| **Хто чекає** | tasks | goroutines |
| **Goroutines** | фіксовані workers | = кількість tasks |
| **Memory** | стабільна | росте з tasks |
| **Use case** | queue consumers | fan-out / resource limit |

Mental model:

```
Worker Pool:  1000 задач, 10 працівників — задачі стоять у черзі
Semaphore:    1000 людей, 10 дверей — всередині може бути лише 10
```

Як вибрати:

```
tasks unbounded / streaming → Worker Pool
fan-out / finite list       → Semaphore
```

## Part 4: Advanced Patterns

### Backpressure

Backpressure — механізм, який **не дозволяє producer генерувати задачі швидше**, ніж система може їх обробляти.

```go
jobs := make(chan Job, 100)  // buffered channel = природній backpressure

for i := 0; i < 10; i++ {
    go worker(jobs)
}

for _, job := range incomingJobs {
    jobs <- job  // блокується якщо buffer заповнений
}
```

```
workers зайняті → buffer заповнюється → producer блокується
```

---

### errgroup + Semaphore

Production патерн: поєднання error propagation + bounded concurrency.

```go
var g errgroup.Group
sem := make(chan struct{}, 10)

for _, url := range urls {
    url := url
    g.Go(func() error {
        sem <- struct{}{}
        defer func() { <-sem }()
        return fetch(url)
    })
}

if err := g.Wait(); err != nil {
    log.Fatal(err)
}
```

Дає:

```
error propagation
bounded concurrency
context cancellation (errgroup.WithContext)
```

---

### Context Cancellation

```go
ctx, cancel := context.WithCancel(context.Background())

for _, job := range jobs {
    go func(job Job) {
        select {
        case <-ctx.Done():
            return
        default:
            process(job)
        }
    }(job)
}
```

```
cancel()
↓
ctx.Done() fires
↓
goroutines завершуються
↓
ресурси звільняються
```

---

### Graceful Shutdown

Worker pool у production повинен підтримувати graceful shutdown.

```go
func worker(ctx context.Context, jobs <-chan Job) {
    for {
        select {
        case <-ctx.Done():
            return
        case job := <-jobs:
            process(job)
        }
    }
}
```

```
stop signal
↓
ctx cancelled
↓
workers завершують поточну задачу
↓
виходять чисто
```

## Production Mental Model

Типовий pipeline Go backend системи:

```
Producer
  ↓
Queue / Buffered Channel  (backpressure)
  ↓
Worker Pool               (bounded goroutines)
  ↓
Semaphore                 (external resource limit)
  ↓
Execution
```

Приклад:

```
Kafka → workers → HTTP calls (обмежені semaphore)
```
