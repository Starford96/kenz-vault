# Go Runtime Internals

Пов'язані нотатки: [[interview/golang/1. Go Concurrency — Complete Guide]]

---

## Part 1: Garbage Collector (GC)

### Що таке GC у Go

Go використовує **тричастковий concurrent mark-and-sweep** збирач сміття.

Головна мета:

```
звільнити пам'ять, яка більше не досяжна
```

GC запускається автоматично і не блокує весь процес.

---

### Три фази GC

```
1. Mark Setup     — зупинка goroutines (STW), увімкнення write barrier
2. Marking        — concurrent сканування всіх досяжних об'єктів
3. Mark Termination — STW, завершення
4. Sweep          — concurrent звільнення недосяжних об'єктів
```

**STW (Stop-The-World)** — короткочасна пауза всіх goroutines.

У сучасних версіях Go STW дуже короткий (мікросекунди).

---

### Tricolor Algorithm

Go позначає кожен об'єкт одним із трьох кольорів:

```
White — не відвіданий (кандидат на видалення)
Grey  — відвіданий, але referents ще не скановані
Black — відвіданий і всі referents скановані
```

Алгоритм:

```
1. Всі об'єкти — white
2. Корені (globals, stacks) → grey
3. Grey об'єкти скануються → black
4. Їх referents → grey
5. Коли grey queue порожня — white = garbage
```

---

### GOGC

`GOGC` — змінна оточення, яка контролює **агресивність GC**.

```
GOGC = 100 (default)
```

Означає: запустити GC коли heap виріс на 100% від попереднього збирання.

```
GOGC = 200  → GC рідше  → менше пауз, більше пам'яті
GOGC = 50   → GC частіше → менше пам'яті, більше CPU
GOGC = off  → GC вимкнений (для тестів)
```

---

### GOMEMLIMIT (Go 1.19+)

`GOMEMLIMIT` — **hard limit** на розмір heap.

```
GOMEMLIMIT=512MiB
```

Коли heap наближається до ліміту — GC стає агресивнішим.

Це запобігає OOM у контейнерах.

```
GOGC=off + GOMEMLIMIT=512MiB
```

Популярний патерн: вимкнути регулярний GC і покластися лише на memory limit.

---

### runtime.GC()

Примусовий запуск GC:

```go
runtime.GC()
```

Використовується у тестах та бенчмарках для отримання стабільних результатів.

## Part 2: Memory Allocation — Heap vs Stack

### Stack vs Heap

```
Stack   — локальні змінні, автоматично звільняються
Heap    — динамічні об'єкти, звільняються GC
```

Go **автоматично вирішує**, куди покласти змінну.

---

### Escape Analysis

**Escape analysis** — компілятор визначає, чи «виходить» змінна за межі функції.

```
змінна не виходить → stack (fast)
змінна виходить   → heap (GC overhead)
```

Приклади:

```go
// stack — не виходить за межі функції
func sum(a, b int) int {
    result := a + b
    return result
}
```

```go
// heap — повертаємо pointer, змінна «виходить»
func newUser() *User {
    u := User{Name: "Alice"}
    return &u
}
```

---

### Як перевірити escape analysis

```bash
go build -gcflags="-m" ./...
```

Виведе:

```
./main.go:10:6: u escapes to heap
./main.go:15:2: result does not escape
```

---

### Memory Allocator

Go використовує власний **tcmalloc-подібний** алокатор.

```
Tiny objects  (< 16 bytes)  → mspan tiny allocator
Small objects (< 32 KB)     → mspan per size class
Large objects (> 32 KB)     → directly from heap
```

Це дозволяє уникати lock contention при алокації.

---

### Stack Growth

Goroutine stack **динамічно росте і зменшується**.

```
початковий розмір stack: ~2 KB
максимум за замовчуванням: 1 GB
```

Коли stack переповнюється:

```
1. новий більший stack виділяється
2. вміст копіюється
3. pointers оновлюються
```

Це називається **stack copying** (або contiguous stacks).

## Part 3: sync Primitives

### sync.Mutex

```go
var mu sync.Mutex

mu.Lock()
// critical section
mu.Unlock()
```

Використовується коли кілька goroutines **читають і пишуть** спільний стан.

---

### sync.RWMutex

```go
var mu sync.RWMutex

// Читачі — можуть бути паралельно
mu.RLock()
// read
mu.RUnlock()

// Письменник — ексклюзивний доступ
mu.Lock()
// write
mu.Unlock()
```

Корисно коли **reads значно переважають writes**.

---

### sync.Once

```go
var once sync.Once

once.Do(func() {
    // виконається рівно один раз
    initDB()
})
```

Патерн: **lazy initialization** у concurrent коді.

---

### sync.WaitGroup

```go
var wg sync.WaitGroup

for i := 0; i < 10; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        work()
    }()
}

wg.Wait()
```

Використовується щоб **дочекатися завершення goroutines**.

---

### sync.Map

```go
var m sync.Map

m.Store("key", value)
v, ok := m.Load("key")
m.Delete("key")
m.Range(func(k, v any) bool {
    // iterate
    return true
})
```

Concurrent-safe map. Підходить для:

```
mostly reads
write-once, read-many
```

Для write-heavy сценаріїв краще .

---

### sync/atomic

```go
var counter int64

atomic.AddInt64(&counter, 1)
v := atomic.LoadInt64(&counter)
atomic.StoreInt64(&counter, 0)
```

Атомарні операції без mutex. Корисно для:

```
counters
flags
lock-free structures
```

CAS (Compare-And-Swap):

```go
atomic.CompareAndSwapInt64(&val, old, new)
```

## Part 4: Go Memory Model

### Happens-Before

Go Memory Model визначає: коли **запис** в одній goroutine **гарантовано видимий** в іншій.

```
якщо A happens-before B
→ зміни зроблені A, видимі для B
```

---

### Synchronization Points

Наступні операції створюють happens-before:

```
channel send → channel receive
sync.Mutex Unlock → Lock
sync.WaitGroup Done → Wait
goroutine start → перша інструкція goroutine
```

---

### Data Race

**Data race** — коли дві goroutines читають/пишуть одну змінну **без синхронізації**.

```go
// ❌ data race
var x int
go func() { x = 1 }()
go func() { fmt.Println(x) }()
```

Результат непередбачуваний.

---

### Race Detector

```bash
go test -race ./...
go run -race main.go
```

Go вбудований race detector виявляє data races **під час виконання**.

```
WARNING: DATA RACE
Write at 0x... by goroutine N
Read at 0x... by goroutine M
```

Overhead: ~5–10x slowdown. Використовується в CI, не в production.

## Part 5: defer Internals

### Як працює defer

```go
func f() {
    defer cleanup()
    // ...
}
```

`defer` — це не просто "виклик у кінці". Під капотом:

```
defer реєструє функцію + аргументи в момент defer
виконує LIFO при поверненні з функції
```

---

### LIFO порядок

```go
defer fmt.Println("1")
defer fmt.Println("2")
defer fmt.Println("3")

// Виведе: 3, 2, 1
```

---

### Аргументи обчислюються одразу

```go
x := 10
defer fmt.Println(x) // запам'ятовує 10

x = 20
// Виведе: 10, а не 20
```

---

### defer + panic + recover

```go
func safe() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("recovered:", r)
        }
    }()

    panic("oops")
}
```

`recover()` перехоплює panic **тільки всередині defer**.

---

### Overhead

defer має невеликий overhead (heap allocation у старих версіях Go).

З Go 1.14 більшість defer компілюється inline — overhead мінімальний.

## Part 6: pprof — Profiling

### Що таке pprof

`pprof` — вбудований профайлер Go.

Дозволяє аналізувати:

```
CPU usage
Memory (heap) allocations
Goroutine count/state
Blocking operations
Mutex contention
```

---

### HTTP endpoint

```go
import _ "net/http/pprof"

http.ListenAndServe(":6060", nil)
```

Потім:

```bash
go tool pprof http://localhost:6060/debug/pprof/heap
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
```

---

### Основні profiles

```
/debug/pprof/heap       — memory allocations
/debug/pprof/profile    — CPU (default 30s)
/debug/pprof/goroutine  — всі goroutines + стеки
/debug/pprof/block      — blocking ops
/debug/pprof/mutex      — mutex contention
```

---

### runtime.ReadMemStats

```go
var m runtime.MemStats
runtime.ReadMemStats(&m)

fmt.Println("Heap alloc:", m.HeapAlloc)
fmt.Println("Num GC:", m.NumGC)
```

Дозволяє читати memory stats програмно.

## Part 7: Netpoller — Async I/O

### Проблема

Класична async I/O (epoll/kqueue) вимагає callback-hell або async/await.

Go вирішує це інакше.

---

### Як працює Netpoller

```
goroutine викликає net.Read()
↓
якщо дані не готові — goroutine паркується
↓
OS thread звільняється
↓
netpoller (epoll/kqueue) чекає на I/O
↓
коли дані готові — goroutine розпаркується
↓
scheduler ставить її в run queue
```

---

### Що це дає

```
синхронний код → async поведінка під капотом
```

Розробник пише:

```go
resp, err := http.Get(url)
```

А runtime робить:

```
non-blocking I/O + goroutine parking + scheduler
```

---

### Чому Go не потребує async/await

```
goroutine блокується → thread не блокується
thread іде виконувати інші goroutines
I/O готовий → goroutine прокидається
```

Це і є причина, чому Go ефективний для network services **без async/await**.

## Quick Reference — Runtime Internals

```
GC:            tricolor mark-and-sweep, concurrent, short STW
GOGC:          heap growth trigger (default 100%)
GOMEMLIMIT:    hard heap limit (Go 1.19+)
Escape:        stack = fast, heap = GC overhead
Stack:         ~2KB start, grows dynamically via copying
Mutex:         exclusive access
RWMutex:       multiple readers OR one writer
sync.Once:     lazy init, thread-safe
sync.Map:      read-heavy concurrent map
atomic:        lock-free counters/flags
Race detector: -race flag, runtime detection
defer:         LIFO, args evaluated immediately
pprof:         CPU / heap / goroutine profiling
netpoller:     epoll/kqueue under hood, no async/await needed
```
