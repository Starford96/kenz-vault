# Harness Engineering: використання Codex у світі agent-first

> Оригінал: [openai.com/index/harness-engineering](https://openai.com/index/harness-engineering/)
> Автор: Ryan Lopopolo, Member of the Technical Staff · 11 лютого 2026
> Подяки: Victor Zhu, Zach Brock та вся команда
> Переклад: лютий 2026

---

Протягом останніх п'яти місяців наша команда проводила експеримент: створення та випуск внутрішньої бети програмного продукту з **0 рядками написаного вручну коду**.

Продукт має щоденних внутрішніх користувачів і зовнішніх альфа-тестерів. Він деплоїться, ламається і лагодиться. Різниця в тому, що кожен рядок коду — бізнес-логіка, тести, CI конфігурація, документація, observability та внутрішній тулінг — був написаний Codex. За нашими оцінками, ми збудували це приблизно в 10 разів швидше, ніж якби писали код вручну.

**Люди направляють. Агенти виконують.**

Ми свідомо обрали це обмеження, щоб побудувати те, що необхідно для збільшення інженерної швидкості на порядки. У нас були тижні, щоб відправити те, що зрештою стало мільйоном рядків коду. Для цього нам потрібно було зрозуміти, що змінюється, коли основна робота інженерної команди — більше не писати код, а проєктувати середовища, формулювати наміри та будувати зворотні зв'язки, які дозволяють агентам Codex виконувати надійну роботу.

Ця стаття про те, що ми дізнались, будуючи абсолютно новий продукт з командою агентів — що ламалось, що накопичувалось, і як максимізувати наш єдиний справді дефіцитний ресурс: людський час і увагу.

## Ми почали з порожнього git-репозиторію

Перший коміт у порожній репозиторій був зроблений наприкінці серпня 2025.

Початковий каркас — структура репозиторію, CI конфігурація, правила форматування, налаштування пакетного менеджера та фреймворк застосунку — був згенерований Codex CLI з використанням GPT-5, керований невеликим набором існуючих шаблонів. Навіть початковий файл AGENTS.md, який вказує агентам, як працювати в репозиторії, сам був написаний Codex.

Не було жодного попередньо написаного людиною коду як якоря системи. З самого початку репозиторій формувався агентом.

П'ять місяців потому репозиторій містить близько мільйона рядків коду — бізнес-логіка, інфраструктура, тулінг, документація та внутрішні утиліти для розробників. За цей період приблизно 1 500 pull request'ів було відкрито та змерджено маленькою командою всього з трьох інженерів, які керували Codex. Це дає середню пропускну здатність 3,5 PR на інженера на день, і що цікаво — пропускна здатність *зросла*, коли команда розширилась до семи інженерів. Важливо, що це не було продуктивністю заради продуктивності: продуктом користувались сотні внутрішніх користувачів, включаючи щоденних активних користувачів.

Протягом усього процесу розробки люди ніколи безпосередньо не писали код. Це стало основною філософією команди: **жодного написаного вручну коду**.

## Переосмислення ролі інженера

Відсутність ручного кодування **запровадила інший вид інженерної роботи, зосередженої на системах, каркасах та важелях впливу**.

Ранній прогрес був повільнішим, ніж ми очікували — не тому, що Codex був нездатний, а тому, що середовище було недоспецифіковане. Агенту бракувало інструментів, абстракцій та внутрішньої структури, необхідних для просування до високорівневих цілей. Основною роботою нашої інженерної команди стало забезпечення агентів можливістю виконувати корисну роботу.

На практиці це означало роботу вглиб: розбиття більших цілей на менші будівельні блоки (дизайн, код, рев'ю, тестування тощо), промптування агента для побудови цих блоків і використання їх для відкриття складніших завдань. Коли щось не працювало, виправлення майже ніколи не було "спробуй старатись більше". Оскільки єдиний спосіб рухатись вперед — це змусити Codex виконати роботу, людські інженери завжди зупинялись і питали: "якої здатності бракує, і як зробити її зрозумілою та обов'язковою для агента?"

Люди взаємодіють із системою майже повністю через промпти: інженер описує завдання, запускає агента і дозволяє йому відкрити pull request. Щоб довести PR до завершення, ми інструктуємо Codex переглянути свої власні зміни локально, запросити додаткові рев'ю від інших агентів як локально, так і в хмарі, відповідати на будь-який зворотний зв'язок від людей або агентів, і ітерувати в циклі, поки всі агенти-рев'юери не будуть задоволені (фактично це [Ralph Wiggum Loop](https://ghuntley.com/loop/)). Codex використовує наші стандартні інструменти розробки напряму (gh, локальні скрипти та вбудовані в репозиторій навички), щоб збирати контекст без того, щоб люди копіювали та вставляли в CLI.

Люди можуть переглядати pull request'и, але це не обов'язково. З часом ми перенесли майже всю роботу з рев'ю на взаємодію агент-агент.

## Підвищення легібельності застосунку

Зі зростанням пропускної здатності коду нашим вузьким місцем стала людська QA-ємність. Оскільки фіксованим обмеженням був людський час і увага, ми працювали над тим, щоб додати агенту більше можливостей, зробивши такі речі як UI застосунку, логи та метрики безпосередньо зрозумілими для Codex.

Наприклад, ми зробили застосунок завантажуваним для кожного git worktree, щоб Codex міг запускати і керувати окремим екземпляром для кожної зміни. Ми також підключили Chrome DevTools Protocol до середовища виконання агента і створили навички для роботи з DOM-знімками, скріншотами та навігацією. Це дозволило Codex відтворювати баги, валідувати фікси та розуміти поведінку UI напряму.

![Codex керує застосунком через Chrome DevTools MCP для валідації своєї роботи](https://images.ctfassets.net/kftzwdyauwt9/1Gu58eNlqDEuITmbqJDmq9/1e2e62f7e15fb16d2da0da5407240564/fig_1__codex_drives_the_app_.png?w=1920&q=90&fm=webp)

Те саме ми зробили з інструментами observability. Логи, метрики та трейси доступні Codex через локальний observability стек, який є ефемерним для кожного worktree. Codex працює з повністю ізольованою версією застосунку — включаючи його логи та метрики, які знищуються після завершення завдання. Агенти можуть запитувати логи через LogQL і метрики через PromQL. З таким контекстом промпти на кшталт "забезпеч, щоб запуск сервісу завершувався менше ніж за 800мс" або "жоден span у цих чотирьох критичних користувацьких шляхах не повинен перевищувати дві секунди" стають реалістичними.

![Повний observability стек для Codex у локальному dev-середовищі](https://images.ctfassets.net/kftzwdyauwt9/4Xr18TZ5G4Bh8zIgsTFIVK/f7ae689ddd8c31664e39d809b0973425/OAI_Harness_engineering_Giving_Codex_a_full_observability_stack_desktop-light__1_.svg?w=1920&q=90)

Ми регулярно бачимо, як окремі запуски Codex працюють над одним завданням понад шість годин (часто поки люди сплять).

## Ми зробили знання репозиторію системою запису

Управління контекстом — одна з найбільших проблем у забезпеченні ефективності агентів для великих і складних завдань. Один з перших уроків був простий: **дайте Codex карту, а не 1000-сторінкову інструкцію.**

Ми спробували підхід "один великий `AGENTS.md`". Він передбачувано провалився:

- **Контекст — дефіцитний ресурс.** Гігантський файл інструкцій витісняє завдання, код та релевантну документацію — тож агент або пропускає ключові обмеження, або починає оптимізувати не ті речі.
- **Занадто багато інструкцій стає *відсутністю інструкцій*.** Коли все "важливе" — нічого не важливе. Агенти починають локально підбирати патерни замість навігації з наміром.
- **Воно миттєво застаріває.** Монолітна інструкція перетворюється на цвинтар застарілих правил.
- **Це важко перевірити.** Один великий блок не піддається механічним перевіркам (покриття, свіжість, відповідальність, перехресні посилання), тож дрейф неминучий.

Тому замість того, щоб розглядати `AGENTS.md` як енциклопедію, ми розглядаємо його як **зміст**.

Знаннєва база репозиторію живе в структурованій директорії `docs/`, яка є системою запису. Короткий `AGENTS.md` (приблизно 100 рядків) вбудовується в контекст і слугує переважно картою.

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```
*Структура знаннєвої бази в репозиторії.*

Проєктна документація каталогізована та індексована. [Архітектурна документація](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html) надає верхньорівневу карту доменів та шарів пакетів. Документ якості оцінює кожен продуктовий домен та архітектурний шар, відстежуючи прогалини з часом.

Плани розглядаються як першокласні артефакти. Складна робота фіксується в [планах виконання](https://cookbook.openai.com/articles/codex_exec_plans) з логами прогресу та рішень. Це забезпечує **прогресивне розкриття**.

Ми забезпечуємо це механічно. Спеціальні лінтери та CI джоби валідують, що знаннєва база актуальна. Регулярний агент "doc-gardening" сканує застарілу документацію і відкриває виправляючі PR.

## Легібельність для агента — головна мета

Оскільки репозиторій повністю згенерований агентом, він оптимізований передусім для *легібельності Codex*. З точки зору агента, все, до чого він не має доступу в контексті — фактично не існує.

![Межі знань агента: що Codex не бачить — не існує](https://images.ctfassets.net/kftzwdyauwt9/7uWHsJIC6o3uQPsnQ2Avz9/8be3e321892054bd215afb2b250a176a/OAI_Harness_engineering_The_limits_of_agent_knowledge_desktop-light.png?w=1920&q=90&fm=webp)

Ми зрозуміли, що потрібно переносити все більше контексту в репо. "Нудні" технології легші для агентів завдяки композабельності та стабільності API. Іноді дешевше перереалізувати функціональність, ніж обходити непрозорі зовнішні бібліотеки.

Перенесення системи у форму, яку агент може інспектувати, збільшує важіль впливу — не лише для Codex, а й для інших агентів (наприклад, [Aardvark](https://openai.com/index/introducing-aardvark/)).

## Забезпечення архітектури та смаку

**Забезпечуючи інваріанти, а не мікроменеджуючи реалізації, ми дозволяємо агентам шіпити швидко.** Наприклад, ми вимагаємо від Codex [парсити дані на межі](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/), але не вказуємо як саме.

Агенти найефективніші зі [строгими межами та передбачуваною структурою](https://bits.logic.inc/p/ai-is-forcing-us-to-write-good-code). Кожен бізнес-домен: Types → Config → Repo → Service → Runtime → UI. Cross-cutting concerns — через Providers.

![Шарова доменна архітектура з явними наскрізними межами](https://images.ctfassets.net/kftzwdyauwt9/4Rlip1H3T9apPlSmWs7Wr8/7708c176bfbe11951e06ad8e2b83bf01/OAI_Harness_engineering_Layered_domain_architecture_with_explicit_cross-cutting_boundries_desktop-light.png?w=1920&q=90&fm=webp)

Кастомні лінти з повідомленнями, що вбудовують інструкції з виправлення прямо в контекст агента. Один раз закодовані — застосовуються скрізь одразу.

Людський смак подається назад у систему безперервно через рев'ю, рефакторингові PR та документацію.

## Пропускна здатність змінює філософію мерджу

Мінімум блокуючих merge gates. Короткоживучі PR. Flaky тести — follow-up ранами. Виправлення дешеві, а очікування — дороге.

## Що означає "згенерований агентом" насправді

Агенти створюють все: код, тести, CI, тулінг, документацію, eval harness'и, рев'ю, скрипти, дашборди. Люди працюють на іншому рівні абстракції.

## Зростаючі рівні автономії

З одного промпта агент тепер може: валідувати стан → відтворити баг → записати відео → реалізувати фікс → валідувати → PR → рев'ю → фікс збірки → ескалювати → змерджити.

## Ентропія та збирання сміття

**Повна автономія також створює нові проблеми.** Codex реплікує навіть субоптимальні патерни. "Золоті принципи" кодуються в репо, фонові завдання сканують відхилення та відкривають рефакторингові PR. Як garbage collection для коду.

## Що ми ще вивчаємо

Що стало зрозумілим: дисципліна проявляється більше в каркасах, ніж у коді. **Наші найскладніші виклики — проєктування середовищ, зворотних зв'язків та систем контролю** для агентів.

[Просто будуйте.](https://openai.com/codex/)
