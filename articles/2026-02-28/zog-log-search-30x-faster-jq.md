# zog — пошук у production-логах у 30x швидше за jq

> **Оригінал:** https://github.com/aikoschurmann/zog
> **Джерело:** r/devops
> **Дата:** 2026-02-21
> **Оцінка:** 7/10
> **Дайджест:** [[articles/2026-02-28/digest|Дайджест 2026-02-28]]

---

Інструмент для пошуку у JSONL-файлах зі швидкістю, обмеженою лише NVMe (3+ ГБ/с). Написаний на Zig, рання стадія розробки.

**GitHub:** https://github.com/aikoschurmann/zog

## Мета

Пошук у production-логах через `jq` буває нестерпно повільним на великих файлах. `grep` швидший, але не розуміє JSON-структуру. Ціль — взяти краще з обох: швидкість grep-рівня з адресацією до JSON-полів.

## Ключові техніки

**SIMD pattern matching** — процесор обробляє 32 байти за інструкцію замість 1. На сучасних x86/ARM це AVX2/NEON-інструкції, що дозволяють сканувати рядки з дуже високою пропускною здатністю.

**Double-buffered async I/O** — поки CPU обробляє один буфер, диск читає наступний. Усуває час очікування вводу/виводу, CPU і диск працюють паралельно.

**Zero heap allocations** — весь скан у попередньо виділених буферах. Жодних GC-пауз, жодного `malloc` під час роботи.

**Pre-compiled query plans** — запит компілюється один раз перед початком сканування. Жодних накладних витрат на парсинг під час виконання.

## Результати

- **30–60x швидше за jq**
- **20–50x швидше за grep**

## Компроміси

- **Немає JSON AST** — не можна відстежувати вкладеність. `{"a": {"b": 1}}` — поле `b` знайде, але не відрізнить від `{"b": 1}` на верхньому рівні.
- **Буквальне числове зіставлення** — `90` і `90.0` — різні рядки для zog.
- **Лише JSONL** — кожен рядок має бути окремим JSON-об'єктом. Pretty-printed JSON не підходить.

Для аналізу application-логів ці обмеження зазвичай прийнятні заради 30–60x приросту швидкості.

## Майбутні ідеї

Автор розглядає постобробку з повним AST-обходом після швидкого SIMD-відбору — щоб поєднати швидкість сканування з точністю повного парсингу.
