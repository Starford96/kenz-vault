# Ключі Google API не були секретами. Але потім з'явився Gemini.

> **Оригінал:** https://trufflesecurity.com/blog/google-api-keys-werent-secrets-but-then-gemini-changed-the-rules
> **Джерело:** Truffle Security
> **Дата:** 2026-02-26
> **Оцінка:** 8/10
> **Дайджест:** [[articles/2026-02-28/digest|Дайджест 2026-02-28]]

---

**TL;DR:** Google понад десятиліття говорив розробникам, що ключі Google API (`AIza...`) — не секрети. Але це вже не так: Gemini приймає ті самі ключі для доступу до приватних даних. Truffle Security знайшли 2 863 живих ключі у публічному інтернеті, спочатку розгорнутих для Maps/Firebase, які тепер також автентифікуються в Gemini. Навіть у самих Google були такі ключі.

## Основна проблема

Google Cloud використовує єдиний формат API-ключів (`AIza...`) для двох принципово різних цілей: публічної ідентифікації та чутливої автентифікації.

Роками Google прямо говорив розробникам, що API-ключі безпечно вбудовувати в клієнтський код. Firebase-чекліст безпеки: «API keys are not secrets». Документація Maps JavaScript: вставляти ключ прямо в HTML.

Це мало сенс — ключі були ідентифікаторами проєктів для білінгу, не обліковими даними.

**Потім з'явився Gemini.**

Коли ви вмикаєте Gemini API у проєкті Google Cloud, наявні API-ключі (включно з тими у публічному JavaScript вашого сайту) можуть **мовчки отримати доступ до чутливих ендпойнтів Gemini**. Без попередження. Без email-сповіщення.

Дві проблеми:

**1. Ретроактивне розширення привілеїв.** Ви створили ключ для Maps три роки тому — саме як радив Google. Минулого місяця хтось у команді ввімкнув Gemini API для прототипу. Ваш публічний ключ Maps тепер є обліковими даними Gemini. Вас ніхто не попередив.

**2. Небезпечні налаштування за замовчуванням.** Новий API-ключ у Google Cloud — за замовчуванням «Unrestricted», тобто дійсний для кожного ввімкненого API в проєкті, включно з Gemini.

Послідовність подій:
1. Розробник створює ключ і вбудовує його для Maps. *(На той момент — безпечний.)*
2. Gemini API вмикається в тому самому проєкті. *(Ключ мовчки отримує доступ до Gemini.)*
3. Розробника ніхто не попереджає. *(Ключ перетворився з публічного ідентифікатора на секретний обліковий запис.)*

## Що може зробити зловмисник

Атака тривіальна: зайти на сайт, подивитися вихідний код, скопіювати `AIza...` ключ з Maps embed. Потім:

```bash
curl "https://generativelanguage.googleapis.com/v1beta/files?key=$API_KEY"
```

Замість `403 Forbidden` — `200 OK`. Далі:

- **Читати приватні дані** — `/files/` та `/cachedContents/` можуть містити завантажені датасети, документи, кешований контекст
- **Нагнати рахунок** — тисячі доларів на день
- **Вичерпати квоти** — повністю зупинити легітимні сервіси Gemini

## 2 863 живих ключі в публічному інтернеті

Truffle Security просканували датасет Common Crawl (листопад 2025, ~700 ТіБ) і виявили **2 863 живих ключі**. Серед жертв — великі фінансові установи, компанії з безпеки, і сам Google.

Один ключ Google: публічно розгорнутий **щонайменше з лютого 2023**, задовго до Gemini. Перевірка `/models` — `200 OK`.

## Таймлайн розкриття

- **21 листопада 2025** — Звіт до Google VDP
- **25 листопада** — Google: «Intended Behavior». Дослідники заперечили.
- **2 грудня** — Google перекласифікував на «Bug»
- **12 грудня** — Google поділився планом усунення
- **13 січня 2026** — Класифіковано як Tier 1: Single-Service Privilege Escalation
- **19 лютого 2026** — Кінець 90-денного вікна, публічне розкриття

## Куди рухається Google

- Нові ключі через AI Studio — тільки Gemini-доступ за замовчуванням
- Блокування скомпрометованих ключів
- Проактивні сповіщення про витоки

## Що робити прямо зараз

**Крок 1:** APIs & Services > Enabled APIs — шукати «Generative Language API». Якщо не ввімкнений — ця вразливість вас не стосується.

**Крок 2:** Якщо ввімкнений — APIs & Services > Credentials. Шукати ключі Unrestricted або з явним дозволом на Generative Language API.

**Крок 3:** Переконатися що жоден такий ключ не є публічним. Старіші ключі — першочергово. Знайшли — ротуйте.
